## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**As an** owner, **I want** an optional deposit or card-on-file hold at booking, **so that** I can reduce no-shows on high-value appointments — without breaching fair-trading or cooling-off rules.

An opt-in deposit/hold taken at booking, applied fairly and suppressed during any cooling-off period (a hard invariant). Off by default; configurable per service. Refund/forfeit follows the cancellation policy.

**Surface:** web (staff) · client · API · **Touches:** deposit/hold at booking, cooling-off suppression, payments provider.

**Build steps (functional):**
- [ ] Opt-in deposit/hold configurable per service (off by default)
- [ ] Take the deposit/hold at booking via the payments provider
- [ ] Suppress the deposit/hold during cooling-off (invariant)
- [ ] Refund/forfeit per the cancellation policy

**References:** PRD-02 rev-4 (deposit/hold, F14 invariant); depends on checkout/payments.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->
- [ ] #1 A deposit/hold is opt-in per service and off by default
- [ ] #2 No deposit/hold is taken while a client is in a cooling-off period
