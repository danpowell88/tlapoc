## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**As a** client, **I want** to create an account with whatever sign-in my clinic offers (social / email+password / one-time code via Entra External ID to start), **so that** I can reach my bookings and records — without the clinic being locked to one identity vendor.

Implements the **identity/auth port** for clients, with an **Entra External ID adapter shipped first** (social, email+password, OTP). Another consumer-identity provider (e.g. Auth0/Cognito/Firebase) is added later as a new adapter with no app changes. Client profiles are tenant-scoped. Used by online booking, intake/consent links and (later) the client app.

**Surface:** client web/app · API · **Touches:** identity port, Entra External ID adapter (first), client profile.

**Build steps (functional):**
- [ ] Client sign-in goes through the identity/auth port, not a hard-coded vendor
- [ ] Ship an Entra External ID adapter first (social, email+password, one-time code)
- [ ] A client profile is created and scoped to the right tenant
- [ ] A second client identity provider can be added as a new adapter without app changes

**References:** PRD-01 §4/§6 (identity), AC3, ADR-0004; sits behind the provider port (TLA-001.13); spike: identity across Flutter and the API.
