## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**As a** compliance officer, **I want** an immutable record of who read or changed sensitive data, **so that** we can prove and investigate access to clinical, medicines and personal information.

The infrastructure for an append-only audit trail: a write path that records who/what/when/tenant for every read and write of protected data, that cannot be edited or deleted after the fact. Feature epics emit events into it; the export/review UI lives in the identity and reporting epics.

**Surface:** API · data · **Touches:** audit event store (append-only).

**Build steps (functional):**
- [ ] An append-only audit store that records who, what, when and tenant
- [ ] A simple, consistent way for any feature to emit an audit event
- [ ] Entries cannot be modified or deleted after they are written
- [ ] Stored entries are retrievable for export by authorised roles

**References:** PRD-01 §6 (audit), C10, AC4.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->
- [ ] #1 A read or write of clinical/medicines/PII data produces an immutable audit entry
- [ ] #2 An audit entry records who, what, when and which tenant
- [ ] #3 No API path can alter or remove an existing audit entry
<!-- AC:END -->
