## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**As an** engineer, **I want** a single payment-provider seam with a Square adapter, **so that** card-present today and recurring autopay later share one PCI-safe integration that never stores card numbers.

A payment-provider abstraction (authorize/capture, refund, void, tokenize, recurring, gift-card) with a Square AU adapter first and cash as an internal tender. No card number (PAN) is ever stored — only provider tokens. The recurring/gift-card capabilities are exercised by the later memberships epic; the seam exists now.

**Surface:** API · **Touches:** payment provider interface, Square adapter, tokenisation.

**Build steps (functional):**
- [ ] A provider interface covering authorize/capture, refund, void, tokenize, recurring, gift-card
- [ ] A Square AU adapter implementing card-present + refund/void now
- [ ] Cash modelled as an internal tender
- [ ] Only provider tokens stored; never a card number

**References:** PRD-06 §6, REQ-PAY-1, ADR-0007, AC5; Square AU recurring spike.
