## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**As an** engineer/owner, **I want** every external dependency to sit behind a swappable port with the concrete vendor chosen by configuration, **so that** we can ship one provider now and change it later without rewriting the app — not every clinic will use the same auth, payments, calendar or accounting vendor.

The foundational ports-and-adapters seam. Each external capability — **identity/auth, payments, calendar, accounting, messaging/SMS, e-prescribing** — is defined as a port (interface) in the core, with one adapter implemented to start and the active provider selected by **per-tenant configuration**. Swapping or adding a provider is a new adapter only; core modules and feature code never change. This is the architectural rule the identity, payments, comms and integrations stories all build on.

**Surface:** API · infra · **Touches:** provider ports/interfaces, per-tenant provider config, adapter registration. Stack-named but vendor-agnostic by design.

**Build steps (functional):**
- [ ] Define a port (interface) per external capability: identity/auth, payments, calendar, accounting, messaging, e-prescribing
- [ ] Select the active provider per tenant via configuration (not hard-coded)
- [ ] Ship exactly one adapter per port to start; document how to add another
- [ ] Core/feature code depends on the port, never on a concrete vendor
- [ ] A second adapter can be added for any port without changing core modules

**References:** ADR-0012 (ports & adapters), ADR-0007 (payments), ADR-0004 (identity); generalises the integration adapter posture (TLA-013) to all external deps.
