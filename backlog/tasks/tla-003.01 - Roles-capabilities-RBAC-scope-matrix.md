## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**As an** owner, **I want** each staff member's role to determine exactly what they can see and do, **so that** people act only within their job and scope of practice.

Role-based access built as capabilities (gate actions) x concerns (drive role-tailored dashboards). Ships the standard roles the prototype validates: NP/prescriber, Lead RN, RN, dermal therapist, reception, owner (read-only clinical/stock), plus the QLD 'designated RN prescriber'. The auth pipeline blocks out-of-scope actions and records the reason.

**Surface:** web (staff) · API · **Touches:** roles, capabilities, authorisation pipeline.

**Build steps (functional):**
- [ ] Standard roles exist with a capability set each
- [ ] Capabilities gate API actions; an out-of-scope action is blocked with a reason
- [ ] A blocked attempt writes an audit event
- [ ] Concerns drive which dashboard cards a role sees (ties to Today)

**References:** PRD-01 §3/§6, C4/C19, ADR-0017; designated-RN-prescriber (QLD Sept 2025).
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->
- [ ] #1 A user attempting an action outside their role's capabilities is blocked with a reason and an audit event
- [ ] #2 The standard role set (incl. designated RN prescriber and read-only owner) is available
